## 🤖 子 Agent 使用准则

**主动拆分任务原则（2026-04-27 设定）**
- **何时创建子 agent**：遇到以下情况时应主动创建子 agent，不要自己硬扛：
  1. 批量处理同类任务（如批量识别 26 张图片、批量分析多个数据源）
  2. 耗时较长的独立任务（如爬虫、报告生成、数据分析）
  3. 可以并行执行的多个子任务
  4. 需要不同模型或 thinking 级别的专项任务
- **主 agent 角色**：负责任务拆分、调度、监督、汇总汇报，不要亲自执行重复性工作
- **子 agent 汇报**：完成后必须向主 agent 汇报结果，主 agent 再汇总给用户
- **定期清理**：定期检查子 agent 状态，清理已完成且久未使用的子 agent，释放资源
- **适用场景举例**：图片批量 OCR、多数据源分析、多报告生成、批量文件处理等

**⚠️ 创建前检查准则（2026-05-07 设定）**
- **规则**：创建子 agent 之前，必须先检查是否已经存在相同功能的子 agent
- **行动**：如果当前已存在相同功能的子 agent（如软件开发工程师子 agent），不要重复创建，直接让已有的子 agent 继续干活
- **检查方法**：通过 `sessions_list` 或 `subagents list` 查看现有子 agent，按 label/功能判断是否已有对应角色

**禁止在高频轮询 API 中使用 subprocess 准则** ⚠️ **血泪教训（2026-04-26）**
- **背景**：在 Dashboard 的 `/api/cron/list` 接口中使用了 `subprocess.run(['openclaw', 'cron', 'list', '--json'])`，而前端每 1 秒轮询一次该接口，导致每秒启动一个 Node.js 子进程（50-100MB），系统内存数分钟内被吃光，直接卡死崩溃
- **规则**：
  1. **修改任何 API 前，必须先检查前端轮询频率**（搜索 setInterval、fetch 等）
  2. **高频轮询（≤5 秒）的 API 绝对禁止使用 subprocess**，只能用纯内存计算（Python 原生库如 psutil）
  3. **如果必须调用外部命令**：加缓存层（至少 30 秒刷新间隔）或改用 WebSocket 推送
  4. **Node.js 子进程尤其危险**：启动整个 V8 运行时，内存占用 50-100MB+，Python 子进程也要 10-30MB
  5. **尊重原代码的设计意图**：如果原代码有意避免某种做法（如注释说明），不要盲目改动，先理解原因
- **适用范围**：任何人使用助手时都适用
- **设定时间**：2026-04-26
- **设定者**：用户085477

## 🛠️ 重要系统发现（2026-04-28）

### ☁️ 云监控服务 (cloudmonitor) 会杀死所有 node 进程
- **问题**：阿里云云监控服务 (`cloudmonitor.service`) 会定期杀死所有 `node` 进程
- **影响**：`sysmon.js` (Node.js 版本) 每 6 秒被杀死一次，导致 CPU/内存监控数据出现 5-6 秒间隔
- **解决**：已将 `sysmon.js` 改写为 Python 版本 (`sysmon_py.py`)，Python 进程不受影响
- **服务文件**：`/etc/systemd/system/openclaw-sysmon.service` 已更新为使用 Python 版本
- **数据文件**：`/root/.openclaw/workspace/tools/sysmon_data.json` 格式不变，Dashboard 兼容

- **秒模式**：显示最近 60 个数据点（约 60 秒），时间戳显示到秒
- **分钟模式**：将秒级数据聚合为分钟级（每 60 秒求平均），显示最近 60 分钟
- **数据来源**：`/api/sysmon-data` 端点读取 `sysmon_data.json`
